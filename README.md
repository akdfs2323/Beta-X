# 📢 Beta X - ระบบกระดานประกาศ (Announcement Board)

โปรเจคย่อยสำหรับการตั้งต้นระบบหน้าฟีดประกาศ แบบ Full-Stack (NestJS + React + PostgreSQL)

---

## 1. วิธีเปิดรันโปรเจคเพื่อใช้งาน (How to run the project)

คุณสามารถเลือกวิธีรันได้ 2 วิธีตามความสะดวกครับ:

### ตัวเลือก A: รันผ่านตัวช่วย Docker (🚀 แนะนำ ง่ายที่สุด!)
วิธีนี้โปรแกรมจะทำงานและจัดการฐานข้อมูลให้เองทั้งหมดด้วยคำสั่งเดียว:
1. เปิดโปรแกรม **Docker Desktop** ไว้บนเครื่อง
2. เปิด Terminal แล้วชี้ไปในโฟลเดอร์รันโปรเจคนี้ (`Beta-X`) 
3. พิมพ์คำสั่งรันเซิร์ฟเวอร์: 
   ```bash
   docker compose up --build -d
   ```
4. รอ 1-2 นาที เมื่อเสร็จแล้วสามารถเปิดเว็บบราวเซอร์ไปที่: 👉 **http://localhost**

*(หากเลิกใช้งาน ต้องการปิดตู้คอนเทนเนอร์: `docker compose down`)*

### ตัวเลือก B: รันแยกทีละส่วนแบบ Manual (Local Development)
กรณีที่ต้องการรันโค้ดสดๆ:
- **สตาร์ท API:** เข้าโฟลเดอร์ `backend` รัน `npm install` ตามด้วย `npm run start:dev` (พอร์ต 3001)
- **สตาร์ทเว็บ:** เข้าโฟลเดอร์ `frontend` รัน `npm install` ตามด้วย `npm run dev` (พอร์ต 5173 แล้วเปิดลิงค์เว็บ)

**🚨 วิธีเริ่มต้นใช้งานหน้าเว็บระดับสิทธิ์แอดมิน (สำคัญ)**
เมื่อเปิดหน้าเว็บครั้งแรกสุด ให้กดปุ่ม **Sign Up** เพื่อสมัครเข้าสู่ระบบ! บัญชีผู้ใช้คนแรกสุดของระบบ จะได้รับสิทธิ์ยศ Admin โดยอัตโนมัติ ซึ่งจะปลดล็อกระบบ "คลิกปักหมุดกระทู้" และ "กดไอคอนถังขยะลบข้อความ" ครับ!

## 2. โครงสร้างเส้นทาง API หลัก (API Endpoint List & Example Requests)

| Method | Endpoint | หน้าที่รับผิดชอบ |
|--------|---------|-------------|
| **POST** | `/api/announcements` | สร้างประกาศใหม่ (บังคับส่ง Token ของระบบล็อกอิน) |
| **GET** | `/api/announcements` | ดึงประกาศทั้งหมด (เรียงโพสต์ที่ Pinned ไว้บนสุดเสมอ) |
| **GET** | `/api/announcements/:id` | ดึงประกาศเจาะจงตัวใดตัวหนึ่งด้วยเลข UUID |
| **DELETE** | `/api/announcements/:id` | ลบประกาศทิ้ง (บังคับส่ง Token เพื่อยืนยันสิทธิ์) |
| **POST** | `/api/auth/register` | สมัครสมาชิกใหม่เพื่อรับประทับตรา JWT Token |
| **POST** | `/api/auth/login` | เข้าสู่ระบบสมาชิกเช็คประวัติ (รับ JWT Token กลับไป) |

## 3. ข้อจำกัดทางสถาปัตยกรรม (Tradeoffs & Assumptions Made)

ในการออกแบบสถาปัตยกรรมตัวนี้ มีการตัดสินใจทำข้อแลกเปลี่ยนในระดับโครงสร้าง 2 ข้อดังนี้: 

1. **Tradeoff ที่ 1: ละทิ้งสายสัมพันธ์ของข้อมูล Author (Denormalization)**
   - *สมมติฐาน:* เนื่องจากโจทย์ระบุเพียงว่าต้องการให้โชว์ "ชื่อคนตั้งโพสต์" บนกระดานข่าว
   - *ข้อแลกเปลี่ยน:* ผมเลือกที่จะละทิ้งการเชื่อมบรรทัด Foreign Key แบบดั้งเดิม (ที่เอาตารางโพสต์ไปผูกกับตาราง User ID) และเลือกเก็บข้อมูลชื่อเป็น String ลงไปตรงๆ ในตารางเลย! สิ่งที่ต้องแลกคือความซ้ำซ้อนของข้อมูลเล็กน้อยตรงฟิลด์ Author แต่สิ่งตอบแทนที่มีค่ามหาศาลคือ "ความเร็วกระฉูด" เพราะฐานข้อมูลไม่ต้องทำ SQL JOIN ข้ามตารางเวลามีผู้ใช้กดเข้าหน้า Feed ส่งผลให้มันรับปริมาณคนใช้งานหน้าโฮมเพจได้ไวประดุจสายฟ้าครับ
   
2. **Tradeoff ที่ 2: การสั่งเปิดออโต้ตั้งค่าโค้ดตาราง `synchronize: true` ให้กับ TypeORM**
   - *สมมติฐาน:* งานนี้เป็นงาน Assignment ที่ต้องการเน้นให้ผู้ตรวจรันขึ้น และรีวิวทดสอบระบบได้ไวที่สุดตั้งแต่ก้าวแรก
   - *ข้อแลกเปลี่ยน:* ผมจงใจเปิดคำสั่งนี้เพื่อให้โครงสร้างตารางและคอลัมน์ถูกเจเนอเรตอัตโนมัติตามโค้ด Entity สิ่งที่สูญเสียไปคือความปลอดภัยของตารางหากเผลอแก้โค้ด ในโลกทำโปรเจคลง Production จริง ผมจะปิดการใช้งานตัวนี้และเปลี่ยนไปใช้ **TypeORM Migrations** อย่างเป็นระบบเพื่อรักษาสถานะ (Version Control) และป้องกันข้อมูลสูญหายแทนครับ

## 4. แผนการอัปเกรดในอนาคต (Future Improvements)

หากมีเวลาปั้นโปรเจคนี้ต่อยอด นี่คือฟีเจอร์ตัวแกร่งที่จะดึงมาเสริมครับ:
1. **ระบบ Load More แบ่งหน้า (Offset Pagination):** 
   หากในอนาคตมีคนโพสต์ข้อมูลเป็นหมื่นๆ ชิ้น การดึงข้อมูลมากองทั้งหมดรวดเดียวจะทำให้เว็บหน่วงและเซิร์ฟเวอร์สำลัก ผมจะเขียนคำสั่ง Offset ไว้ที่ฐานข้อมูล (`Skip`/`Take`) ควบคู่กับท่าลากเมาส์เลื่อนหน้าจอลงบนเว็บและโหลดต่ออัตโนมัติ (Infinite Scrolling) ดึงข้อมูลมาทีละ 20 โพสต์ก็เกินพอครับ
2. **Optimistic UI Updates (การล่อลวงตา):** 
   จะทำการอัปเกรดหน้าบ้านในจังหวะ "กดลบข้อมูลบนกระดาน" ให้มันตอบสนองหลอกตาเสมือนว่าโพสต์โดนลบกระเด็นหายไปในเสี้ยววินาที! ทิ้งคำสั่งแอบโยนข้อมูลให้หลังบ้านจัดการตามยถากรรมแบบไม่ต้องรอโหลด หมุนสปินเนอร์ ทำให้ผู้ใช้รู้สึกว่าโปรแกรมลื่นปรื๊ดดั่งสายน้ำครับ
